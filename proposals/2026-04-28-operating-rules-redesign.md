## 2. なぜなぜ (メタ・なぜルールが守られなくなるか)

| Why | 答え |
|---|---|
| Why1 | CLAUDE.md が長く、セッション開始時の精読が形骸化する |
| Why2 | 現役ルールと過去なぜなぜ分析が同一ファイルに混在し、重要度の差がフラットに見える |

## 3. 仕組み的対策 (4 つ)

### A. ファイル分割 (責務ごとに分ける)

| ファイル | 役割 | 上限 | 更新タイミング |
|---|---|---|---|
| **RULES.md** (新設) | 現役ルールのみ。汎用 + 各プロジェクト共通 | **200 行** | 上限超過時に統廃合 (CI で block) |
| **LESSONS.md** (新設) | なぜなぜ分析アーカイブ (append-only) | なし | 事象発生時。RULES.md からは見出しリンクのみ |
| **PROJECT_STATE.md** (P003 用) | プロジェクト現況スナップショット | なし | タスク完了時 |
| **CLAUDE.md** | 起動手順 + 上記 3 ファイルへのリンクのみ | **80 行** | RULES.md 改訂時のみ |
| TASKS / WORKING / HISTORY | 既存どおり | - | - |

CLAUDE.md は「セッション開始でやること」と「次に読むファイル」だけに削る。

### B. 「絶対ルール 7 か条」(汎用・暗記可能サイズに圧縮)

1. **完了の定義 = 本番動作確認済み**。`done.sh <id> <verify_target>` を必ず通す。verify_target 省略は warn 表示
2. **依存口頭化**: ファイル変更前に「これを参照している箇所」を 1 行明文化。書けなければ変更しない
3. **同名ファイル並行編集禁止**: WORKING.md 記載が無い状態でコードに 1 行も触らない
4. **PII / secrets コード直書き禁止**: env var or Secrets Manager 必須
5. **再発防止 = フルセット必須**: Why1〜5 + 仕組み対策 3 つ以上 + **物理ゲート 1 つ以上** (CI / pre-commit / 外部 SLI のいずれか)。テーブル 1 行追記は再発防止ではない
6. **外部観測 SLI 必須**: ユーザーが見るデータの鮮度・正常性を、コード外から測るモニタを 1 つ以上持つ
7. **長文化禁止**: RULES.md が 200 行を超えたら、必ず統廃合または LESSONS.md 移動

> **ポイント**: 7 は「ルールが増えたら他を削れ」というメタルール。これが無いと A の上限が守られない。

### C. 物理ゲート (スクリプト 4 本)

| スクリプト | 役割 | 起動 |
|---|---|---|
| `scripts/check_rules_size.sh` | RULES.md > 200 行で commit を reject | pre-commit hook |
| `scripts/done.sh` (既存・改修) | verify_target 必須化。`url:` / `lambda:` / `topic-ai:` を強制 | タスク完了時 |
| `scripts/freshness_check.sh` | 主要 API の `updatedAt` ズレ > 90min で Slack 警告 | EventBridge 1h |
| `scripts/rotate_lessons.sh` | RULES.md 内の「なぜなぜ」セクションを LESSONS.md へ自動退避 | 半月次 cron / 手動 |

### D. なぜなぜテンプレート (LLM が真似する用)

LESSONS.md の冒頭に固定テンプレートを置く:

```
### <事象名> なぜなぜ分析 (YYYY-MM-DD 制定)

**起きたこと**: 1-2 文で事実

| Why | 答え |
|---|---|
| Why1 | … |
| Why2 | … |
| Why3 | … |
| Why4 | … |
| Why5 | … |

**仕組み的対策** (3 つ以上、うち 1 つ以上は物理ゲート):
1. **コードルール追加** — RULES.md に … (CLAUDE.md ではなく RULES.md)
2. **CI / pre-commit 物理ブロック** — …
3. **外部観測 SLI** — …
```

これに従わない再発防止 commit は CI で reject (commit メッセージ + diff から検出)。

## 7. 補足: なぜ「ルールを増やす」ではなく「構造を変える」のか

Claude (LLM) の挙動として、規則は **量ではなく到達可能性** で守られる。

- 規則が多くて到達できない → 守られない
- 規則が少ないが必読フォルダにある → 守られる

これまでの追加方針 (新事象ごとに表へ 1 行 + なぜなぜ 20 行追加) は、人間運用なら有効だが LLM では「セッション開始時の読み取り深度」に律速される。**ファイルサイズが上限を超えた瞬間、形骸化が連鎖する**。

ads.txt 事件のなぜなぜ忘れ・T218 の作業滞留・T242 の鮮度ズレ気付き遅れは、いずれも「該当ルールが書かれている場所まで届いていない」結果。だから今回は **追加ではなく圧縮と分離** を提案する。
