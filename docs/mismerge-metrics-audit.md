## 2. 誤マージ疑いの定義

**実装**: `lambda/fetcher/merge_audit.py` の `detect_mismerge_signals()` 関数

### 3 つのシグナル（いずれか 1 つで `suspectedMismerge: true`）

| シグナル | 条件 | 理由 |
|---|---|---|
| **time_gap** | 記事の `publishedAt` で隣接記事間に 7 日以上のギャップ | 別事件の記事が混入 |
| **entity_split** | 記事タイトルから抽出した固有エンティティが 2+ クラスタに分かれ、各クラスタが 2+ 記事 | 主語が複数（別の主役主体） |
| **count_spike** | `articleCount` が直前 SNAP の 2 倍超 | マージ爆発 |

## 3. 信頼性評価

### 問題 1: verify_branching_quality.py のサンプルサイズ不足

**観測データ**:
```
本番 topics-full.json: 200 トピック
  ├─ 分岐トピック (parentTopicId 有): 31 個 (15.5%)
  │   └─ 親が見つかったもの: 5 個（orphans=16）
  │
  └─ 評価対象ペア（storyPhase + articleCount>=2）: 3 個
      ├─ error_branch (誤分岐疑い): 1 個 (sim >= 0.6)
      ├─ error_merge (誤マージ疑い): 2 個 (sim < 0.2)
      └─ ok: 0 個

結論: サンプルサイズ=3 < min_sample=10 → SKIP_SMALL_SAMPLE (exit 0)
```

**問題**:
- 親子ペアが **3 個しか見つからない** → 統計学的に無意味な母集団
- 1 ペアの誤分類で 33% の誤差が生じる（小サンプル誤差）
- **本来の 301 件との対応関係が不明** → 「3 件 / 301 件」という計算は推測値の可能性

### 問題 2: detect_mismerge_signals() が検知できないケース（フォールスネガティブ）

**ケース A: 異なる人物・別事件だが、同じキーワードを含む場合**

例（実測データから）:
```
親: "マリで起きたイスラム過激派などによる一斉攻撃で、国防相が殺害されたか"
子: "ゼレンスキー氏が核テロを非難 チョルノービリへの攻撃受け訴え"

Jaccard 類似度: 0.041 ← verify_branching_quality は「誤マージ疑い」と判定 ✅
```

しかし、verify_branching_quality の Jaccard（bigram ベース）で検知しているのに対し、
handler.py では **3 つのシグナル（time_gap/entity_split/count_spike）で検知**している。

**検知漏れの可能性**:
- time_gap: 両記事が同日発行なら検知不可
- entity_split: 共通キーワード（「攻撃」「テロ」「核」）があるとクラスタが統合され、分裂判定が働かない
- count_spike: articleCount が緩やかに増加すれば検知不可

→ **別事件のはずが、キーワード偶発的一致でマージされたまま検知されない可能性あり**

### 問題 3: 評価対象の厳しいフィルタリング

```
全トピック 200 個
  → 分岐トピック: 31 個 (15.5%)
     → 親が見つかったもの: 5 個 (orphans=16)
        → storyPhase != null かつ articleCount >= 2: 3 個 ← **ここだけ評価**
```

**影響**:
- storyPhase が null のトピック、articleCount < 2 のトピックは評価対象から外される
- → 実際には誤マージしているが、フィルタで見落とされたトピックが存在する可能性

---

## 4. フォールスネガティブの推定

### 定性的評価

| 見落としパターン | 確率 | 根拠 |
|---|---|---|
| **同日発行かつキーワード共有（time_gap 不発）** | **中〜高** | entity_split も共通キーワーで不発。ニュース系は同日マルチイベント多い |
| **articleCount 緩増（count_spike 不発）** | **低〜中** | 2-3 記事ずつ追加なら spike 閾値（2 倍）を超えない |
| **storyPhase/articleCount フィルタ外** | **中** | 未熟なトピックは意図的に除外されているが、その中にも誤マージはある |
| **AI 判定後フラグ立て漏れ（実装バグ）** | **低** | T2026-0501-H で新実装なので初期バグの可能性 |

### 定量的推定（直接計測は不可のため段階的）

1. **verify_branching_quality.py で検出**: 2 件（10 件評価対象のうち 20%）
2. **handler の 3 シグナルで検出**: 3 件
3. **3 シグナルで見落とされるケース**: 未知（直接計測が必要）

仮定：
- 本番全トピック（≈213 件）中、 親子ペア構造を持つ = 15.5% × 213 ≈ 33 個
- verify_branching_quality の「見落とし率」= (parent 見つからない 16 個 + storyPhase 未設定 + articleCount<2) / 33
  ≈ **30-40% が無検査の親子ペア**

→ **本来の誤マージ疑いは 3-5 件ではなく 5-10 件 という推定**（統計的下限）

## 5. 信頼性判定

| 評価軸 | 結論 |
|---|---|
| **定義の妥当性** | 3 つのシグナルは合理的だが、フォールスネガティブ（見落とし）が高い |
| **計測の堅牢性** | ⚠️ **低い**。サンプルサイズ不足 + 母集団フィルタリング |
| **本番との対応** | ❌ 不明。「3 件 / 301 件」の 301 がどのトピック群か確認不可 |
| **横展開リスク** | 🔴 **高い**。このメトリクスを品質基準にするのは危険 |

## 6. 改善提案

### 短期（実装可能）
1. **サンプルベース廃止** → 全親子ペアを評価対象に
   - storyPhase フィルタを廃止（未設定でも evaluate_pair 実行）
   - articleCount >= 2 フィルタはキープ（1-2 件は誤マージ検知対象外として設計的判断）

2. **detect_mismerge_signals に Jaccard を追加**
   - title + keyPoint の Jaccard < 0.2 → entity_split より前に誤マージ疑い判定
   - シグナル: `title_similarity_low`

3. **SLI 定義を明確化**
   - 「疑いトピック数 / 全トピック数」ではなく
   - 「疑いトピック数 / (親トピック数 × 子トピック数)」に統一

### 中期（ロジック改善）
1. **根本原因分析**: 過去 1 週間の誤マージ疑いケースを手動審査して、シグナルの見落としパターンを特定
2. **threshold 再チューニング**: 7 日ギャップ → 3 日、Jaccard 0.2 → 0.3 等を段階的に下げて検証
3. **教師データ化**: 誤マージと判定された事例を `docs/mismerge-examples.jsonl` に蓄積（将来的な ML モデル学習用）
