## ビジョン

**「世界がわかるようになる、数分で」**

ニュースの羅列ではなく、出来事の「文脈の地図」を提供する。縦の因果（大項目→中項目→小項目）と横の波及（ジャンル横断）がつながり、ユーザーが世界の構造を自分ごととして理解できる。

## 設計原則（崩さないこと）

1. **精度が命** — 誤った因果関係は信頼を壊す。確信度低は表示しない
2. **単発記事は出さない** — 記事2件以上・整合性チェック通過のみ
3. **粒度は「具体的な一つの出来事」単位**
4. **表現は断定より示唆**（「可能性がある」「指摘されている」）
5. **ユーザーを主語にする** — 「この出来事が自分の生活にどう影響するか」
6. **実装したら必ずブラウザで確認** — コードだけで完了としない
7. **面白おかしく + 一発理解** — 事実は正確に・論理展開の違いはエンタメ的に対比

## 中核哲学: なぜ Flotopic か (2026-04-27 ナオヤ言語化)

### ニュース理解のフレームワーク

ニュースを理解するには **6 段階** が要る (システム開発と同じ):

```
1. 背景      ← 「なぜそれが存在するか」「前提条件」
2. 課題      ← 「何が問題か」
3. 目的      ← 「何を達成したいか」
4. 要件      ← 「条件・制約」
5. 手段      ← 「具体的に何をするか」
6. 設計      ← 「どう作るか / 制度・枠組み」
7. 結果      ← 「どうなったか」
8. 今後      ← 「どうなるか」
```

**既存ニュースは 5-8 (手段以降) しか書かない**。「協議した」「合意した」「批判した」だけで、なぜそれをしているか (1-3 = 背景・課題・目的) は前提知識として読者に押し付ける。**結果として途中参加ユーザーは『何の問題を解決しようとしている話か』が分からない**。

### Flotopic がやること

**A. 抜けてる 1-3 (背景・課題・目的) を AI が補う**
- aiSummary を「背景→課題→目的→手段→結果→今後」フレームで内部整理してから書かせる
- 「主語+動詞+目的語+目的(何のために)」を必須化
- 「協議」「会談」「合意」「停戦」「批判」等の動詞には『〜をめぐる』『〜に向けた』形式で目的を明示

**B. 5-8 (手段以降) のメディア間ズレを暴く**
- 背景・目的は事実なのでメディアでズレない
- ズレるのは「何を強調するか・どう着地させるか・どう論理展開するか」
- perspectives フィールドで比較フォーマット表示: 「{社A}は『{側面A}』で{結論A}に着地、{社B}は『{側面B}』で{結論B}に着地」
- これがメディアと SNS の両方に対する Flotopic の独自価値

**C. 出力は『面白おかしく + 一発理解』**
- 事実ベースでトピックをまとめる時、論理展開の違いを「ただ平均化して薄くする」のではなく **対比として魅せる**
- 教科書的な分析より、読み物として刺さる切り口を優先
- 「実は〜」「まるで〜のよう」「ところが〜」のような物語的接続詞を活用 (ただし事実は正確に)
- ユーザーが 1 トピック 30 秒で「あ、こういう話か」と腹落ちできることがゴール

### この哲学を AI プロンプトに反映した commit
- `448e7f6` aiSummary に「背景→課題→目的→手段→結果→今後」フレーム導入 (全3モード)
- `4dcbd9e` perspectives を「手段以降のメディア間ズレを暴く」に強化
- 既存の core 機能 (background / spreadReason / forecast / outlook / topicTitle / topicLevel / parentTopicTitle / relatedTopicTitles) と組み合わせて、6段階フレーム全体を表現する

## トピック詳細ページ 設計原則 (2026-04-28 確定)

### 基本思想
- **注目度**（みんなの関心）と**内容のポイント**（コンテンツ）は完全に別の情報。混在させない
- 「なぜ今注目か」はAIが語るのではなく、グラフ（記事数・注目度スパイク）がデータで示す
- 「後から読んでも1発でわかる」ことが最重要。背景込みの時系列ストーリーで構造化する

### トピック詳細ページ構成（確定）

```
[注目度ゾーン]
  状況ラベル: 発端 / 進行中 / 沈静化 / 決着
  注目度グラフ（記事数推移） ← "なぜ今か"はこれが答える

[コンテンツゾーン]
  状況解説（keyPoint）: 背景込みの時系列ストーリー、200-300字、箇条書き禁止
  各社の見解（perspectives）: メディア名付き
  これからの注目ポイント（watchPoints）: ①②③形式、断言ではなく観察点
  AI予想（outlook）: 検証可能な仮説 + 後から判定バッジ（pending/matched/partial/missed）
```

### AIフィールド設計原則
- `keyPoint`: 1週間後に初めて読む人が背景〜現在を一気に理解できるストーリー。「もともと〇〇があり→△△が起き→現在□□」の流れ。言葉を選んでキレのある日本語で
- `statusLabel`: 発端/進行中/沈静化/決着
- `watchPoints`: 「ここを見ておくといい」複数の観察点。断言・予言ではない
- `outlook`: AI予想として明示。後から記事が来たとき matched/partial/missed で判定される

### ストーリー間の関連付け原則
- **ストーリー分岐・繋がりは「内容（セマンティック）」で判断**
- 注目度の数字（記事数・スコア）で関連トピックを決めない
- 登場人物・因果関係・共通テーマ・エンティティの重複で判断する

### Bluesky投稿選定原則
- 注目度スコア（velocityScore）の高いトピックを優先
- 記事数だけで選ばない（Google Trendsが不安定な場合はhybridスコアで補完）
- 直近4件に同一topicIdがある場合はスキップ（重複防止）
