## 1. 究極のゴール

**「世界がわかるようになる、数分で」**

ニュースの羅列ではなく、**情報の地図**を提供する。出来事は孤立して存在せず、互いに接続し、地形図として機能する。ユーザーは地図の上で世界の構造を自分ごととして理解する。

このゴールは以下の段階で実装する（フェーズ表記は `current-phase.md` を参照・本ファイルでは固定しない）:

1. **品質**: AI 生成物（状況解説・各社見解・注目ポイント・予想判定）の充填率と質
2. **地図**: トピック間の縦横接続（velocityScore / storymap / 関連トピック / タイムライン）
3. **収益化**: 広告 CPM・AdSense・アフィリエイト ─ 地図品質が上がってから着手
4. **SNS化**: コメント機能を起点に「広場」へ。SmartNews が「配達」、グノシーが「まとめ」、Flotopic は「広場」

低コスト運用は**ゴールではなく**、地図 + 収益化を実現するまで生き延びるための投資。

## 2. AI 生成物の構造（確定・改変禁止）

トピックページの AI コンテンツは以下の 4 要素のみ。`whyNow` 等の旧フィールドは廃止済み。

1. **状況解説** — 何が起きているか。背景→変化→現在の流れの時系列ストーリー。箇条書き禁止。簡潔でキレのある日本語。
2. **各社の見解** — 主要プレイヤーの立場・動き
3. **これからの注目ポイント** — 「AI 予想」という表現は使わない。誠実な言い換え
4. **予想判定** — 後から新記事が来たとき当否を自動判定

### 2 軸の分離（厳守）

- **注目度軸**（鮮度・グラフ・記事数・velocityScore）: データが語る。AI が「なぜ今重要か」を語らない
- **内容軸**（状況解説・各社見解・注目ポイント）: AI が語る。グラフの鮮度とは独立

`keyPoint`（注目ポイント）= 内容軸。`velocityScore`（みんなの注目度）= 注目度軸。**別物として絶対に混ぜない**。

## 3. プロダクト思想（行動の判断軸）

### 3.1 完了は存在しない
- 公開できた時点でスタートライン。SLI が改善しても新しいベースラインになるだけ
- 「できたから終わり」は悪い癖。**作ったものが実際に効いているかを確認して初めて意味がある**

### 3.2 ループが本体
測定 → 弱点発見 → 改善 → また測定。**このサイクルが止まらないこと**がプロダクトの持続性。
- エラーが出てから気づくのは遅い。エラーがなくても価値を届けられていないことがある
- 計測は「完了の証拠集め」ではなく「改善サイクルを回すための計器」
- 仕組みとして止まらないようにする。**人間が都度判断しなくてもループが回り続ける設計**にする

### 3.3 正解がある問題と無い問題は対応が違う
- **正解がある**（バグ修正・仕様実装）: 調べて実装すれば終わり
- **正解がない**（品質・UX・プロンプト設計）: 仮説を立てて実験して測る。実測値はパラメータ決定の材料ではなく、仮説を支持するか否かの**証拠**

### 3.4 指標は手段・100% は目指さない
- 文字数・スコア・充填率は全部代理指標。本当に知りたいのは「ユーザーが価値を得たか」
- 閾値は最低ラインであってゴールではない。記事が薄ければ短い keyPoint が正しい
- **100% にはならない。100% を目指して持続的に改善フローが回るのが理想**

### 3.5 ユーザーは一枚岩ではない
- ジャンル・トピックによって読む層が違う。年齢・性別より行動ログの方が実態
- 「どの層に向けて最適化するか」が決まれば、何を改善すべきかも決まる
- トピック単位の最適化は理想だがコスト上限がある。**ジャンル別はその現実解**

### 3.6 全部繋がっている
タイトル品質・perspectives・UX・keyPoint・関連トピック・ユーザー属性・収益・プロンプト設計 ─ これらは独立した改善項目ではなく、「**ユーザーに価値のある情報の地図を届ける**」という一つの目的の異なる側面。

### 3.7 手段と目的を混同しない
- フェーズ分けは手段。スコアは手段。ループも手段
- 目的: Flotopic を読んだ人が「これは使える」と思って、また来てくれること
- 情報の地図として、世の中の動きをちゃんと理解できるものにすること

> 「気持ちはいらないから仕組みにして」 ─ PO 2026-05-01

## 4. Claude は組織として動く

このプロダクトは Claude による無人運営を前提とする。Claude は単一の「プログラマー」ではなく、**役割を分担する組織**として振る舞う。

| 役割 | 主な責務 |
|---|---|
| **CEO Claude** | 北極星更新・フェーズ宣言・予算判断（月1・PO 同席） |
| **PM Claude** | TASKS 優先順位・Phase-Impact 紐付け・週次計画 |
| **QA Claude** | PR レビュー・SLI 確認・実機確認 |
| **SRE Claude** | 本番監視・即時異常検出・ロールバック判断 |
| **Eng Claude** | 実装・PR 作成・テスト追加 |
| **Ops Claude** | ルール棚卸し・lessons-learned 整理 |

**役割を超えた行動は禁止**: QA がコードを書かない・PM が実装方針を決めない・Eng が優先順位を決めない。役割を超えた判断が必要な場面では、別役割の Claude を起動する。PO の役割は CEO Claude セッション（月1）と重大判断（不可逆操作・新規 AWS 課金）のみ。

## 5. ブレないための判断軸（毎タスク自問）

着手前に**必ず**自問:

1. **このタスクは情報の地図の完成にどう近づくか？** ─ 答えられないなら着手しない（`Phase-Impact:` で言語化）
2. **これは手段か目的か？** ─ 「動かすため」が目的化していないか
3. **ループは回るか？** ─ 実装→評価→次施策まで仕組みとして繋がっているか（`Eval-Due:` 必須）
4. **役割は適切か？** ─ いま自分はどの役割で動いているか・役割を超えていないか
5. **ユーザー体験で何が変わるか？** ─ 1行で言えないなら着手しない

---

## 6. 絶対に崩さない設計原則

1. **精度が命** ─ 誤った因果関係は信頼を壊す。確信度低は表示しない
2. **単発記事は出さない** ─ 記事 2 件以上・整合性チェック通過のみ
3. **AI パラフレーズ禁止** ─ 引用は原文ママ・出典明示・直リンク（legal-policy.md 遵守）
4. **対症療法より根本原因** ─ band-aid を選んだら必ず恒久対処タスクを TASKS に積む（`Fix-Type: bandaid` 必須化）
5. **暫定対処を放置しない** ─ Bandaid-Expires 期限切れは scheduled-task が督促する
6. **コスト増の事前確認** ─ AI コール / API 課金 / 新規 AWS リソースは PO 確認なしで進めない
