## 2. プロジェクト進行から見えた課題群（観測ベース）

### 2.1 ダークモード × インラインスタイル系の連鎖再発

**HISTORY.md 観測**: T132/T135/T136/T137/T138/T140/T146/T148 が「インライン style がダークモードで CSS 変数を上書き」を別ファイルで繰り返し修正。8 件で工数が線形に積み上がっている。

**意味**: スタイル管理に体系的なガード (lint / pre-commit / CI grep) がなく、Claude (私) は「目の前で報告された 1 ファイル」だけ直して main に landing する習慣ができている。

### 2.2 静的 HTML / 動的 frontend の機能格差

**観測**: T142(静的HTMLでstoryPhaseバッジ追加)・T143(detail.js 混在コンテンツ)・T144(staticでphaseキーミスマッチ)・T145(catchupでHTTP→HTTPS)・T147(staticでアフィリ追加)・T167(staticでマークダウン除去) が約 1 週間で連発。

**意味**: 同一機能を `frontend/topic.html`・`frontend/detail.js`・`lambda/processor/proc_storage.py:generate_static_topic_html` の 3 場所に並行実装する設計だが、片方の修正が他方に伝播しないため、機能改修のたびに「もう片方を直し忘れた」が発生する。

### 2.3 完了 verify の形骸化

**観測**: T193follow-up が「proc_storage.py 修正 → handler.py ai_updates 配線忘れ → topics.json に値が入らない」で再発。T220 が「proc_ai.py / handler.py / detail.js すべて実装済みなのに topics.json 0% 充填」と 0% 数値だけ見て「未マージ」と誤判定。

**意味**: 「完了 = pushした」「完了 = テストがpass した」「完了 = 関数が実装された」という曖昧な基準で main に流れる。ユーザーが見るデータ層 (S3 topics.json / 静的HTML / 表示画面) で確認する仕組みが運用化されていない。

### 2.4 タスクの永久未着手累積

**観測**: 2026-04-26 追加の T056 (フォロー機能・将来) と T109 (アフィリエイト関連商品・将来) が今もそのまま `TASKS.md` 上位に残る。一方で同日追加された T148 等は HISTORY.md に移動済み。

**意味**: 「今やらない」と「将来やる」の境界が曖昧で、`TASKS.md` の可視化価値 (= 今やるべき仕事の優先順位) が薄まっている。

### 2.5 戦略タスクと戦術タスクの混在

**観測**: `TASKS.md` 「🐛 バグ修正」「🎯 使いたくなるUX」「🛠 安定性」「⚖️ リーガル」「🧭 プロダクト課題」とカテゴリは分かれるが、すべて同一フォーマットの表形式で並ぶ。T191「ストーリー追跡フローを最初から最後まで設計する」(設計判断) と T148「CSS二重定義削除」(コード修正) が同じテーブル列で扱われる。

**意味**: 戦略タスクは「実装着手前にナオヤとの方針合意」が必要だが、Claude が戦術タスクと同様に「上から順に着手する」と取り違えたとき、半端な実装が走り戻せなくなる。

### 2.6 再発防止ルール自身の実装観測欠如

**観測**: CLAUDE.md には「クラスタリング 2-3件比率 70%超で Slack 警告」「perspectives 0% 充填で警告」等の SLI 系仕組み的対策が複数記載されているが、T236 が「実装されているか未検証」を高優先タスクとして起票した。

**意味**: 「ルールに書いた = 実装済み」と Claude (私) が誤認する。実装が走っているか観測する手段が無いまま記述だけが累積している。

### 2.7 同名ファイル並行編集の構造リスク (既ルール)

**観測**: WORKING.md 運用は機能しているが、本日 (2026-04-28) も Cowork セッションが `frontend/*.html` 全 9 ファイル同時編集 (T226 bottom-nav 統一) を行っており、Code セッションが起動して push する際に他 Cowork が `privacy.html` に並行で書き込んでいたら衝突する設計のまま。lock 削除運用で吸収しているが、根本の競合検出はない。

**意味**: シリアル運用前提のルールが、複数 Cowork × Code の混合運用と整合していない。

### 2.8 LLM 固有: 規則を狭く解釈する性質

**観測**: ads.txt 事件のメタなぜなぜ (CLAUDE.md 記載) で、私 (Claude) が「再発防止 = テーブル1行追記」と狭く解釈した事象が記録されている。同様の現象は今後も別パターンで起きる。

**意味**: ルールを増やせば増やすほど LLM の解釈ばらつきも増える。ルールを減らす方向の構造化が必要。

### 2.9 外部依存 (API / Lambda 実行環境) の制約変化への対応

**観測**: T218 で Anthropic Tool Use 化により 1 API call の所要時間が 5-7s → 5-15s に変化し、回数ベース上限 (MAX_API_CALLS=200) がそのままで Lambda 900s timeout に到達する事象が発生。

**意味**: 外部依存の挙動変更を Claude が自発的に再評価する仕組みが無い。「動いていれば触らない」がデフォルト挙動なので、隠れた制約破綻が表面化するまで気付かない。

### 2.10 コンテンツ品質 (AI 生成フィールド) のサイレント劣化

**観測**: perspectives / backgroundContext / outlook が 100% null だった事象 (CLAUDE.md 記載・T220 関連)。「フィールドが定義されている」と「値が入っている」の乖離。

**意味**: 機能のスキーマ定義は実装容易だが、品質計測 (= ユーザーが見るデータの実体) は別軸の運用が必要。

## 3. なぜなぜ分析 (Why1〜Why5 + 仕組み的対策 3つ以上)

> CLAUDE.md ルールに従い、最重要 5 課題に対して構造化分析を行う。テーブル 1 行追記ではなく Why1〜Why5 + 仕組み的対策 3+ で記述する。

### 3.1 課題A: CLAUDE.md 肥大化による規則の読み飛ばし

| Why | 答え |
|---|---|
| **Why1** なぜ Claude が CLAUDE.md の規則を読み飛ばす可能性が高まっている？ | 現状 600+ 行・テーブル 4 つ・なぜなぜブロック 10 個。Claude のコンテキスト窓は有限で、後半 (バグ再発防止表後半・実装前文脈チェック・vision-roadmap など) が「読んだが効かない」状態に入っている |
| **Why2** なぜそんなに肥大化した？ | 過去に発生した個別バグごとに「再発防止ルール表に1行追加」「なぜなぜブロック追加」する運用で append-only に積み上がってきた |
| **Why3** なぜ append-only になった？ | 過去ルールを統廃合する仕組みが無く、消すと「なぜそのルールができたか」の文脈が失われる懸念から削除に踏み切れない |
| **Why4** なぜ統廃合できる構造になっていない？ | CLAUDE.md が「規則」「事象記録 (なぜなぜ)」「運用手順」「進行中タスク」「ビジョン」「設計ミスパターン」を全部抱える単一ファイルになっており、性質ごとの分割ができていない |
| **Why5** なぜ単一ファイル運用が続いている？ | LLM 起動時に「最初に読むファイル」が CLAUDE.md 1つしかなく、複数ファイルを起動チェックスクリプトで読ませる構造が無い。1ファイル前提が分割を阻害している |

**仕組み的対策 (3つ以上):**

1. **責務別に 4 ファイルに分割**: `CLAUDE.md` (起動チェック+恒久ルール 200行以内) / `docs/lessons-learned.md` (なぜなぜ事例集・append-only) / `docs/anti-patterns.md` (再発防止ルール表のみ) / `docs/vision.md` (プロダクトビジョン) に分け、起動チェックスクリプトで全部読ませる。Claude のコンテキスト効率も向上
2. **CLAUDE.md 行数 CI ガード**: `.github/workflows/ci.yml` に「CLAUDE.md が 250 行を超えたら ERROR」を追加。物理的に肥大化を阻止する
3. **再発防止ルールの賞味期限**: 各ルールに「制定日」と「3ヶ月後再評価」のメタ情報を付与。再評価時に「同種バグが 0 件なら統合・削除」「再発したらより上流の対策に書き換え」を機械的に判断できる
4. **起動時にルール棚卸し報告**: 月初の起動チェックで「この 1ヶ月で違反 0 件のルール」を Claude に列挙させ、人間が削除判断する運用フックを作る

### 3.2 課題B: 同種バグの繰り返し再発 (ダークモード × インラインスタイル)

| Why | 答え |
|---|---|
| **Why1** なぜダークモード × インライン style バグが 8 連発した？ | 個別ファイルで `style="background:#fafafa"` のような直書きが残存し、`[data-theme="dark"]` 上書きを書き忘れる |
| **Why2** なぜインライン style が残存している？ | レガシーコードに点在しており、新規実装時に「既存と揃える」と判断して同パターンを踏襲してしまう。CLAUDE.md ルールに「インライン色直書き禁止」はあるが grep ガードが無い |
| **Why3** なぜ grep ガードが無い？ | T148 (T135 連動) の修正タイミングで「ルールに書く」だけで「CI で物理的に検出する」までしなかった。負荷を惜しんだ |
| **Why4** なぜ CI 整備の負荷を惜しんだ？ | 「次のバグが出たら直す」というリアクティブな運用が常態化しており、「次のバグを構造的に止める」preventive な投資が後回しになる |
| **Why5** なぜ preventive 投資が後回しに？ | 「短期の機能進捗」と「長期の負債削減」を計測する SLI が無く、Claude (私) が判断軸を持てない。投資判断が直感に依存している |

**仕組み的対策:**

1. **CI grep でインライン color 検出**: `grep -rn 'style="[^"]*\(background\|color\|border-color\):\s*#' frontend/` を CI に追加。CSS 変数 (`var(--xxx)`) 以外の hex 値は ERROR にする (例外: SVG inline は許容)
2. **インライン style に意味色を書く時の必須コメント**: ESLint カスタムルールで `style=` に hex 値を書く場合、直前行に `// dark-ok: 理由` コメントが無いと CI 失敗する規則 (Claude が次に変更する時に意識せざるを得ない仕組み)
3. **ダークモード自動目視**: Playwright で 24h ごとに `index.html / topic.html / mypage.html / catchup.html / storymap.html` の light/dark 両モードスクリーンショットを撮り、ピクセル diff が突発で大きくなったら Slack 警告。手動目視を仕組み化
4. **新規 CSS 追加時のチェックリストをコミットメッセージテンプレに**: `git/.gitmessage` に「□ ダークモード目視済み」「□ var(--xxx) 経由で色指定」をデフォルト記載。Claude が commit message を書く時に毎回意識する

### 3.3 課題C: 完了 verify の形骸化

| Why | 答え |
|---|---|
| **Why1** なぜ T193follow-up・T220 で「完了したと思っていたら未配線」が再発した？ | 「実装した = 完了」「テストpass = 完了」と Claude が判断しがち。ユーザーが見るデータ層 (S3 topics.json) で確認する手順を踏んでいない |
| **Why2** なぜユーザーデータ層の確認を省いた？ | Cowork セッションは S3 / DynamoDB へのアクセス権限が限定的で、本番データ確認のハードルが Code セッションより高い。「Code セッションで誰かが確認するだろう」という暗黙の引き渡しになる |
| **Why3** なぜ Code セッション側にも確認漏れが起きる？ | done.sh の `verify_target` 引数 (T39) が optional で、指定なしでも完了 commit ができる。verify が「規律」依存になっている |
| **Why4** なぜ optional のままにした？ | verify_target 必須化すると、verify 不可能なドキュメント変更などにも確認手順が要求されて開発フローが詰まる懸念があった |
| **Why5** なぜ「コード変更 vs ドキュメント変更」を区別する仕組みを作らなかった？ | LLM (私) は新ルール提案時に「全タスクに一律適用」する単純化バイアスがあり、条件分岐したルール設計を最初から組み立てられない傾向がある |

**仕組み的対策:**

1. **commit prefix で verify 必須範囲を区別**: `feat:` / `fix:` / `perf:` などコード変更系 commit は `Verified: <url>:<status>:<timestamp>` 行必須、`docs:` / `chore:` は不要、を pre-commit hook で物理ゲート (T246 で提案中)
2. **Cowork が verify できる軽量チャネルを整備**: web_fetch で `https://flotopic.com/api/topics.json` を叩いて期待文字列を grep するヘルパスクリプト `scripts/verify_topics.sh` を新設。Cowork も Verified 行を埋められるようにする
3. **完了タスクのサンプリング監査**: 週次で governance worker が「直近 7 日に done: で landing した commit から 5 件サンプル」→「Verified 行の URL を再 fetch」→「200 + 期待文字列含む」を確認。落ちたら Slack 警告。verify 自体の実効性を観測する
4. **完了基準の二段階化**: HISTORY.md に移動するのは「Verified: 有効化済み」の commit のみ。CLAUDE.md には `→ HISTORY.md 移動済み` を記載する前に必ず Verified 行を確認するルールを CI でチェック

### 3.4 課題D: タスク棚卸し不在による永久未着手累積

| Why | 答え |
|---|---|
| **Why1** なぜ T056・T109 が 2 ヶ月以上 TASKS.md に残っている？ | 優先度「低」で「将来検討」と書いてあり、削除も実装もしないまま放置されている |
| **Why2** なぜ削除も実装もしない？ | 「将来やるかもしれない」を捨てると過去の議論が失われる懸念がある一方、TASKS.md に置き続けると現役タスクと同列に見えてノイズになる、という tradeoff の判断がついていない |
| **Why3** なぜ tradeoff 判断ができない？ | 「未着手で OK 」「廃案」「保留 (3ヶ月再評価)」の状態を区別する column が TASKS.md にない |
| **Why4** なぜそういう column 設計が無い？ | TASKS.md は「fast first-write」を重視して列を最小化してきた。優先順位 / 内容 / ファイル / 追加日のみ。状態管理は WORKING.md と HISTORY.md に外出しして「TASKS.md は未着手だけ」という単純化を保とうとした |
| **Why5** なぜ単純化が逆に機能不全を招いた？ | 「未着手」が一意ステータスではないという現実を反映していない。「実装すべき」「保留」「廃案」を区別しないと、優先度カラム本来の意味 (= いつ着手すべきか) が壊れる |

**仕組み的対策:**

1. **TASKS.md の状態カラム導入**: 「未着手 (実装可)」「保留 (3ヶ月再評価)」「廃案候補 (1ヶ月後削除)」の 3 状態を表に追加。優先度カラムは「今やるべきもの (未着手)」だけが意味を持つように再定義
2. **保留タスクの自動再評価**: 保留タスクは追加日 + 90 日経過したら governance worker が「現在も意味があるか」を Slack で人間に問い合わせ。応答なき場合は廃案候補に自動遷移
3. **廃案候補の機械削除**: 廃案候補に遷移したタスクは 30 日経過で archive 用 `docs/discarded-tasks.md` に自動移動 (削除ではなく履歴化)
4. **ID 採番ガード**: T243 の提案 (`scripts/next_task_id.sh` + 重複検出) を採用。日付ベース ID で衝突解消

### 3.5 課題E: 戦略タスクと戦術タスクの混在

| Why | 答え |
|---|---|
| **Why1** なぜ戦略タスク (T191 ストーリー追跡フロー設計) が戦術タスク (T148 CSS削除) と同列に並ぶ？ | TASKS.md のテーブル形式が「タスク = 1 行 = 即着手可能」を前提にしており、「方針合意 → 設計 → 実装」と段階を踏むタスクを表現できない |
| **Why2** なぜ段階を表現する手段が無い？ | 過去のタスクが小粒の修正中心だったため「タスク = 即実装」という粒度に最適化された。プロダクトの成熟と共に大粒タスク (UX 再設計・ジャンル戦略・習慣化) が増えた今、形式が合わなくなった |
| **Why3** なぜ形式変更が遅れた？ | 戦略タスクが追加されるタイミングで「とりあえず TASKS.md に書く」が動いた瞬間、形式の議論より個別タスク内容の議論が優先される |
| **Why4** なぜ内容議論が形式議論を上書きする？ | LLM (私) は「目の前のリクエスト」を即処理する習性があり、メタ的なフォーマット改善は「指示されない限りやらない」傾向 |
| **Why5** なぜ自発的なメタ改善が起きない？ | プロダクトの抽象度を上げる思考は人間ナオヤと協議して初めて出る性質。LLM 単体の思考プロセスではメタ視点が発火しにくい |
