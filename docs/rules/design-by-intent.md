# 設計意図の記録ルール（恒久ルール）

全ての設計判断は以下の4点をセットで記録する。記録なしの feat:/perf: commit は hook で reject。

1. **目的レイヤー**: L1(最上位プロダクト目的) / L2(運営目的) / L3(手段目的) のどれに対応するか
2. **想定効果（数値）**: どうなることを期待するか。根拠付き
3. **成功基準（閾値）**: 何を以て達成とみなすか
4. **変更トリガー**: 上位目的が変わった時にこの決定をどう見直すか

目的が上位層からズレた手段は、いくら最適化しても問題を悪化させる。

## 記録テンプレート

`docs/decisions/NNN-<slug>.md` を新規作成して以下の構造で記録する。

```markdown
# NNN: <決定タイトル>

## 目的レイヤー
- L1（最上位）: <プロダクトの最終目的>
- L2（運営）: <運営上の目的>
- L3（本決定）: <この決定が直接解決する問題>

## 上位目的との競合リスク
- <この決定が L1/L2 を毀損するリスク>
- 対策: <リスクを緩和する仕組み>

## 決定
<何を決めたか>

## 想定効果（数値）
- <指標>: <期待値> （根拠: <根拠>）

## 検証方法
<どのログ・指標でいつ確認するか>

## 成功基準（閾値）
| 指標 | 閾値 | 判定方法 |
|---|---|---|

## 変更トリガー
- <上位目的が変わった場合にこの決定をどう見直すか>
```

## commit-msg での強制

`feat:` `perf:` commit には `Expected-Effect:` 行が必須（commit-msg hook で reject）。
`Expected-Effect:` には以下のいずれかを書く:

```
Expected-Effect: <指標> <期待値> (docs/decisions/NNN-<slug>.md)
Expected-Effect-Skip: <理由>  # build artifact / test fixture / lint fix など
```

`fix:` `chore:` `docs:` は不要。

## 運用ルール
- `docs/decisions/` のファイル番号は 001 から連番（重複禁止）
- 既存の決定を覆す場合は既存ファイルを更新（削除不可・append ベース）
- L3 の決定が L1/L2 と方向が一致しない場合、実装前に PO に確認する
