## コンテンツ引用ポリシー（2026-04-28 確定）

### 引用の根拠

- **著作権法32条の引用**を根拠として、ニュース記事のスニペット（1〜2文）を出典明示つきで表示する
- 引用は法的に認められた行為であり、許諾を取らずに行ってよい（ただし要件を満たすこと）

### 引用要件（必須）

| 要件 | 実装 |
|---|---|
| **出典明記** | 「〇〇（NHK等）より引用」とソース名を本文中に明示 |
| **直リンク** | 元記事への直リンク必須。リンク切れ防止のため正規URLを保持 |
| **改変禁止** | 引用文は原文のまま。誤字脱字も含めて改変しない |
| **主従関係** | 自分のコンテンツが「主」、引用部分が「従」になるよう量を抑える（1〜2文程度） |
| **AIパラフレーズ禁止** | AI生成での言い換えに差し替えない（かえって印象が悪く、引用要件も満たさなくなる） |

## クロール・取得ポリシー（2026-04-28 確定）

### 基本方針

- **robots.txt を遵守する**（Disallow 指定パスにアクセスしない）
- **User-Agent は正直に名乗る**（例: `flotopic-bot/1.0`）。偽装しない
- **スニペット取得は RSS フィード経由を優先**（直スクレイピングより規約的に安全）
- **一次情報ドメインからの直スクレイピングは最小限に留める**（必要時のみ、頻度制御つき）

### 実装上の注意

- RSS が提供されているソースは RSS を最優先
- 直アクセス時はリクエスト間隔を空ける（連続叩き禁止）
- ソース側から拒否された場合（403/429）は即座にバックオフ＋当該ソースを一時除外

## 一次情報バッジの設計根拠（2026-04-28 確定）

### 設計思想

- **一次情報バッジ + 直リンクは出版社への送客になる**ため、むしろ歓迎される設計である
- 自前で完結させて送客しない設計は、引用元から見ると「持ち出し」に映るので避ける
- ソロ開発者として訴訟リスクは取れないため、**グレーより白寄りの設計を徹底する**

### 判定ロジック（実装ルール）

| ルール | 内容 |
|---|---|
| **ドメインで判定** | ソース名の文字列マッチではなく、実URL の eTLD+1 を見る |
| **ホワイトリスト方式** | 信頼できる一次情報ドメインを明示列挙して照合（NHK, 主要新聞社, 公的機関等） |
| **偽装URL を弾く** | `nhk.evil.com` のようなサブドメイン偽装は eTLD+1 が `evil.com` なので一次情報判定しない |
| **HTTPS 必須** | http:// は一次情報バッジ対象外（中間者改ざんリスク） |

## まとめ：判断に迷ったとき

1. **白か黒か迷ったら白寄り**を選ぶ。グレーは取らない。
2. **送客になる設計**か？ 出版社が見て「うちにトラフィックが流れてくる」なら歓迎される。
3. **引用要件を物理的に満たしているか**？ 出典・直リンク・原文ママ・主従関係の4点を実装でガードする。
4. **規約・robots.txt に書かれていないからOK**ではなく、**書かれていなくても誠実に振る舞う**。
