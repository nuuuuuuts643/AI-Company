# Flotopic 機能フェーズ定義（2026-05-03 制定）

> **背景**: 実装済みだが位置づけが曖昧な機能が多い。大工事（keyPoint品質改善・about修正・コメント機能整備など）の方針を揃えるため、機能フェーズを明文化する。
> **影響範囲**: about.html FAQ 書き換え・機能開発優先順位・PR description の Phase-Impact 記述

---

## フェーズ定義（4段階）

### Phase A: コアバリュー（今すぐ・ログイン不要）

**開始条件**: 今すぐ（実装済）

**機能リスト**:
- トピック一覧・詳細閲覧
- AI分析（4要素構造: 状況解説 / 各社の見解 / 注目ポイント / 予想判定）
- フェーズラベル（🌱始まり → 📡広まってる → 🔥急上昇 → 📍今ここ → ✅ひと段落）
- カテゴリフィルター
- 関連トピック・ストーリー分岐
- 推移グラフ・記事タイムライン
- 朝刊・夕刊モデル（1日2回更新: JST 05:30 / 17:30）

**方針**:
ここを完璧にすることが最優先。コンテンツが弱いうちは何をログインさせても無意味。**フェーズ2が完了（keyPoint充填率70%）するまでは、Phase B/C/D の機能は凍結。**

---

### Phase B: 軽いエンゲージメント（信用が生まれたら・ログイン不要で提供）

**開始条件**: コンテンツ品質が安定・リピーターが存在すると確認できたとき

**機能リスト**:
- お気に入り（ローカルストレージ方式でログイン不要）
- 閲覧履歴の改善

**方針**:
摩擦を最小化。ログイン不要でも便利を実感させる。Phase A の信用が十分でない段階でログイン導入は逆効果。

**現状**: お気に入り機能は実装済みだが、ログイン必須になっており摩擦が高い。Phase A 安定後に LS 化に改める予定（別タスク）。

---

### Phase C: アイデンティティ層（～500 visits/day・信用確立後）

**開始条件**: 
- Daily visits 500超え、かつコンテンツ品質に自信が持てるとき
- **コメント削除機能が実装済み**（このフェーズで初めて導入）

**機能リスト**:
- Googleログイン（「デバイスをまたいで履歴・お気に入りを同期」という明確な価値を提示してから）
- コメント投稿（削除機能を実装してから初めて提供）
- マイページ（サーバー側履歴・お気に入り同期）

**方針**:
「なぜログインするか」の価値が明確になってから導入する。コメント削除機能がない状態での提供は不可。

**現状**:
- Googleログイン: 実装済みだが、Phase C 条件（500visits・コメント削除）未達で提供が時期尚早
- コメント投稿: 実装済みだが、削除機能がないため GDPR的問題あり。削除機能実装まで改修不可
- マイページ: 実装済み・概ね正常

---

### Phase D: SNS化（コミュニティができてから）

**開始条件**: 定常的なコメント投稿者が存在し、コミュニティの萌芽がある状態

**機能リスト**:
- フォロー通知（お気に入りトピックの動きをプッシュ）
- ユーザー間の反応
- パーソナライズ
- 通知機能

**方針**:
ユーザー数が増えてから実装。先に作ると誰もいない広場になる。

## about.htmlへの反映方針

このフェーズ設計に基づき、about.htmlのFAQを以下のように整理する：

### 更新内容

1. **「今すぐ使える機能」セクション** → Phase Aの機能のみ記載。ログイン不要を明示。
   - 記載: トピック閲覧・AI分析・グラフ・フィルター・タイムライン
   - 削除: ログイン機能（Phase C）・コメント機能（Phase C）・通知（Phase D）

2. **「開発中・今後追加予定」セクション** → Phase B〜Dの機能を「準備中」として正直に記載
   - 「お気に入りはローカルストレージ化を検討中」
   - 「コメント削除機能実装後に正式提供予定」
   - 「ユーザーコミュニティが成長してから通知機能の開発を検討」

3. **削除**:
   - 「振り返る」機能の記述（既に削除済み）

4. **修正**:
   - AI更新頻度「1日4回」→「1日2回（朝5:30・夕17:30 JST）」
   - フェーズ説明「発端→拡散→...」→実際のemojiラベル説明に統一（`docs/product-direction.md` AI生成物の構造と連動）
