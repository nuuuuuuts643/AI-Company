# 001: ソースフィルタリング設計

## 目的レイヤー
- **L1（最上位）**: ニュースを読まない人が世の中を地図として把握できる
- **L2（運営）**: 信頼できる・偏りのない情報を持続可能なコストで届ける
- **L3（本決定）**: 転載・重複・低品質記事を除外してClaudeコストを削減する

## 上位目的との競合リスク
- フィルタが厳しすぎる → 重要な単独スクープを除外 → L1「網羅的な地図」が毀損される
- 対策: `single_source` フラグで単独報道を捨てずに保持し、Claude 指示に「単独報道」注記を入れる

## 目的（プロダクトゴールとの接続）
ニュースを読まない人が信頼できる地図を手に入れるために、
Claude に投げる記事の質を上げてコストを下げる。
低品質・転載・重複記事を混ぜても精度は上がらず課金だけ増える。

## 決定
`proc_sources.py` で記事をフィルタしてからチャプター生成に渡す。
フィルタ内容は `source-policy.json` に外部定義（コード外で変更可能）。

## フィルタ内容（優先度順）
1. ソースティア: Tier-A(一次ソース) > Tier-B(信頼報道) のみ対象
2. 転載検出: 「〇〇によると」「（共同）」「（ロイター）」パターン → 元記事あれば除外、なければ1件残す
3. 内容重複除去: 同内容記事は1件のみ（文字数+ソースティアで選択）
4. 最小文字数: min_content_length 未満を除外
5. ソース偏り防止: 同一ソースから最大N件
6. 単独報道フラグ: 1ソースのみ → Claude指示に「単独報道」を追記

## 却下した代替案
- 全記事を Claude にスコアリングさせる: 追加API費用が本末転倒
- ソース数でトピックの信頼性を判定する: 単独スクープを排除してしまう

## 想定効果（数値）
- **入力トークン削減: 平均 30〜40% 削減見込み**
  - 根拠①: 国内ニュースの転載率は一般的に 30〜50%（共同・時事配信が複数メディアに転載）
  - 根拠②: 同一トピックで 10 件集まると内容が重複する記事が平均 3〜5 件（内部観察）
  - 根拠③: Tier-A/B 以外の記事（まとめサイト等）は質が低くトークンの無駄
  - 保守的見積もり: フィルタ除去率 30% × トークン比例仮定 → 約 30% 削減
  - 楽観的見積もり: 除去率 40% + 重複除去でさらに 10〜15% → 計 40〜55% 削減

## 検証方法
CloudWatch Logs Insights で `[SOURCE_FILTER]` ログを集計する。

```
filter @message like /\[SOURCE_FILTER\]/
| parse @message "before=* after=*" as before, after
| stats avg(before) as avg_before, avg(after) as avg_after, count() as runs by bin(1d)
```

**確認タイミング**: deploy 翌日（2026-05-05）に 24h 分のログを確認。
**確認項目**:
1. `before` と `after` の差分が平均 2 件以上あること
2. `single_source=True` のトピックが存在し「単独報道」フラグが付与されていること
3. エラーログに `proc_sources` 関連の例外が出ていないこと

## 成功基準（閾値）
| 指標 | 閾値 | 判定方法 |
|---|---|---|
| 入力トークン削減率 | **20% 以上** | CloudWatch `[SOURCE_FILTER]` の before/after 比 |
| チャプター品質スコア | **現状維持（悪化なし）** | p003-sonnet 週次評価で判断（2026-05-12 以降） |
| エラー率 | **0%**（filter_articles で例外なし） | CloudWatch Errors メトリクス |

削減率 20% 未満の場合は `source-policy.json` の `max_articles_per_source` を 3 → 2 に絞り込む。
チャプター品質が悪化した場合は `dedup_syndicated: false` で転載検出を無効化して原因を特定する。

## 変更トリガー
- 「深く読みたい人向け」にプロダクト目的が広がった場合 → max_per_source を緩和
- 特定ジャンルで一次ソースが少ない場合 → Tier-B の基準を見直す
- 削減率 20% 未満が 7 日続いた場合 → max_per_source を 3 → 2 に強化
