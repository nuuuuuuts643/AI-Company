## 目的レイヤー（L1 > L2 > L3）

| レイヤー | 内容 |
|---|---|
| **L1（存在意義）** | ニュースを普段読まない人が「世の中で何が起きているか」を地図として理解できる |
| **L2（品質基準）** | 信頼できる情報を偏りなく・持続可能なコストで届ける |
| **L3（実装手段）** | 縦軸（ストーリー時系列）と横軸（関連トピック・元記事）の2軸で情報を構造化する |

**注意**: L3（2軸ナビ）は L1/L2 を達成するための手段。L1 が果たされない実装変更（例: 深すぎてわかりにくい・情報過多）は方向性が正しくても L3 として誤り。

---

## 現状の問題（困り事）

- 現在の詳細ページは「keyPoint + perspectives + outlook + 記事リスト」を並べているだけ
- **読者の問い**: 「この話、どこまで進んだの？」「今どういう状況？」→ 答えられない
- **読者の問い**: 「この記事はどのストーリーの話？」→ 答えられない
- 記事の羅列は「世界を地図として見る」体験を提供できない

---

## 解決する設計（2軸双方向ナビゲーション）

### 軸1: ストーリーライン（縦軸・時系列）

チャプター型で事件の進行を時系列に見せる。

```
┌─────────────────────────────────────┐
│ トピック: 米中関税摩擦               │
├─────────────────────────────────────┤
│ 📅 2026-05-03                       │
│ ▶ [要約] 米国が追加関税25%を発表     │  ← summary (事実のみ)
│ ▶ [予測] 中国は報復関税で即応する見込み │  ← prediction (断定形)
│ [詳しく見る ▼]                      │
│   [解説] 国内製造業への影響は...     │  ← commentary (1ライナーテーザー)
│   [元記事 3件]                      │  ← articleIds
├─────────────────────────────────────┤
│ 📅 2026-04-28                       │
│ ...                                  │
└─────────────────────────────────────┘
```

### 軸2: 関連マップ（横軸・トピック間）

- 各チャプターの `commentary` 内に `[topicId:xxxx]` プレースホルダー → フロントがタップ可能リンクに変換
- `relatedTopicIds` から横方向の関連トピックカードを表示
- 記事要約カードからも「このストーリーへのバックリンク」を提供

### 双方向リンク構造

```
ストーリー詳細ページ
    ↓ commentary 内リンク
関連ストーリー詳細ページ

記事要約カード（将来実装）
    ↓ 「このストーリーで取り上げられています」
ストーリー詳細ページ
```

## 画面設計（Step 6 S3 実装対象）

### A. トピック詳細ページ（`detail.js` / `detail.html`）

**現状から変えること:**
- `outlook` / `perspectives` のベタ置き表示 → チャプター時系列リスト表示に置き換え
- 記事リスト（全記事羅列） → 各チャプターの展開式に移動

**変えないこと:**
- ヘッダー（トピック名・カテゴリ・更新日）
- 関連トピックセクション（`relatedTopicIds` 利用）
- 盛り上がり度・記事増加トレンド（`articleTrend`）

**チャプターカードのUI:**
```
[ 日付バッジ ] [ summary 1-3行 ]
[ prediction 1行 — インライン表示 ]
[ 詳しく見る ▼ ] ← タップで展開
  └ commentary（`[topicId:xxx]` はリンクボタンとして描画）
  └ 元記事 N件（タイトル + URL）
```

**commentary テーザー（折りたたみ時）:**
- commentary の最初の1文（〜50文字）を常時表示
- 「続きを読む」でフル展開
- L1 目標: 「気になったら深掘り、気にならなければサラッと流せる」

### B. トピック一覧ページ（変更なし）

- 既存の `keyPoint` / `articleTrend` / `updatedAt` 表示を維持
- チャプターがある場合、`lastChapterDate` を「最終更新」として活用（追加のみ・変更なし）

### C. 記事要約カード（将来フェーズ・今は設計のみ）

**目的**: 各記事に「この記事はどのストーリーで取り上げられているか」を示す

**前提**:
1. フェッチャーが DynamoDB に保存している記事データの内容確認が必要（title/URL/descriptionのみか、full contentもあるか）
2. processor が記事初回遭遇時に要約を1回生成 → DDB `article_summaries` テーブルに保存
3. チャプター生成でも同じキャッシュを参照（多重呼び出し禁止）

**今は設計だけ記録。実装は記事要約キャッシュ設計（003番）を先に行う。**

## 想定効果（測定可能な形で事前定義）

| 指標 | 現状 | 目標 | 測定方法 |
|---|---|---|---|
| 詳細ページ平均滞在時間 | 測定なし（要計測） | +30% 増 | CloudFront ログ or tracker |
| 詳細ページ「詳しく見る」クリック率 | 0%（機能なし） | 10% 以上 | tracker Lambda にイベント追加 |
| 関連トピックへの遷移率 | 測定なし | +20% 増 | tracker |
| 1訪問あたりページビュー数 | 測定なし（要計測） | +15% 増 | CloudFront ログ |

**成功基準（最低ライン）:**
- 「詳しく見る」クリック率 5% 以上（10% 未達でも 5% 以上なら継続）
- 滞在時間が下がらない（新UIで離脱が増えていないこと）

**失敗基準（切り戻しトリガー）:**
- 滞在時間が -20% 以上減少
- エラー率 1% 以上（チャプターデータ欠損による表示崩れ）
- `chapters` フィールドが存在しないトピックでレイアウト崩れ

## リスク・競合リスク

| リスク | 詳細 | 対策 |
|---|---|---|
| チャプターなしトピックの表示 | `chapters` が空配列の場合、画面が空になる | `chapters` 空の場合は既存 `keyPoint` + `perspectives` にフォールバック |
| commentary 過多で離脱 | 情報量が多すぎてL1目標（サラッと理解）を阻害 | テーザー表示（50文字）で制御・展開は任意 |
| `[topicId:xxx]` リンクの未解決 | 参照先トピックが削除・非表示の場合 | リンクを描画しない（サーバー側でバリデーション）か、ただのテキストで表示 |
| `lastChapterDate` なしトピック | 旧データは `chapters` も `lastChapterDate` もない | chapters 空のフォールバック適用 |

## L1/L2 との整合チェック（実装前に必ず確認）

- [ ] チャプターなしトピックでも画面が崩れない → L1 担保
- [ ] commentary を開かなくても「何が起きたか」は summary + prediction で分かる → L1 担保
- [ ] `[topicId:xxx]` リンクが地図的な回遊を実現している（単なる「関連リンク」でなく文脈の中にある）→ L1 担保
- [ ] 記事の重複・低品質記事が表示されない（S2.5 ソースフィルタが効いている）→ L2 担保
- [ ] Claude 追加呼び出しなし（commentary は processor 生成済み）→ L2 コスト担保

## 実装順序

1. **フォールバック実装から始める**: `chapters` が空の場合は既存表示に fallback → 壊れないことを確認
2. **チャプターカード表示**: `chapters` あり の場合だけ新UI表示
3. **展開式 commentary**: タップで開閉
4. **`[topicId:xxx]` リンクレンダリング**: 正規表現で置換
5. **関連トピックカード**: `relatedTopicIds` から横軸表示
6. **tracker イベント追加**: クリック率測定のため「詳しく見る」タップをトラッキング
7. **本番確認**: `flotopic.com` で 2〜3 チャプターあるトピックを目視確認

**禁止**: S2.5 のソースフィルタ効果確認（翌日 CloudWatch 確認）前にS3コードを本番適用しない。効果確認後に S3 PR を作成する。

## 変更トリガー（この設計を変えるべき状況）

- 滞在時間 -20% 以上 → チャプター型ではなく既存表示に戻す
- `chapters` 充填率 30% 未満のまま 2週間経過 → S2 processor の生成ロジックを確認
- commentary のクリック率 5% 未満 → テーザー文言・UIを改善する前に、そもそも commentary の内容が読む価値があるか確認（L1 との整合見直し）
- POから L1 の定義変更指示 → この文書全体を書き直してから実装に着手
