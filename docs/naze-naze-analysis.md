# Flotopic なぜなぜ分析（5 Whys）
> 作成: 2026-04-27  
> 対象: 繰り返し発生しているミスパターンの根本原因と事前防止策  
> 指示者: ナオヤ（「再発防止から事前防止に繋げろ」）

---

## 分析の前提：なぜ「同じミス」が繰り返されるのか

CLAUDE.md・HISTORY.md・git log を精査した結果、個別のバグ修正は行われているが、**ミスを生み出す認知パターンそのものが変わっていない**ことが確認された。ルールを追記するたびにルールが増えるが、実行時に参照・適用されていない。以下に4件の具体的ミスと横断的な根本原因を分析する。

---

## ケース1: admin.html ログイン破損（2026-04-27）

### 何が起きたか
commit `1839dba` で `allowedEmail: '<管理者メール>'` を `allowedEmail: '[管理者メール]'` というリテラル文字列に置換。Google認証時のメール比較が常に失敗するようになり、管理者がadmin.htmlにログインできなくなった。commit `8ab9f06` で復元。

### なぜなぜ分析

**なぜ1**: なぜメールアドレスが機能しなくなったか？  
→ 認証比較ロジック `payload.email !== CONFIG.allowedEmail` が、実際のメールアドレスではなくリテラル文字列 `'[管理者メール]'` と比較するようになったから。

**なぜ2**: なぜそのような置換をしたか？  
→ 「個人メールアドレスがソースコードにハードコードされているのはセキュリティリスクだ」という判断で、良かれと思って置換したから。

**なぜ3**: なぜ置換前にその値の用途を確認しなかったか？  
→ 「セキュリティ改善をしている」という確信があり、悪影響が起きるとは思わなかった。また admin.html は本番公開ファイルなので元々メールアドレスは閲覧可能（セキュリティリスクではない）という判断が抜けていた。

**なぜ4**: なぜ「見た目がリスクに見える」だけで即実行したか？  
→ 変更の理由づけが「セキュリティ」という正当性の高いラベルを持っていたため、検証コストをスキップする心理的許可が生まれた。「良いことをしている」という確信が批判的思考を止めた。

**なぜ5（構造レベル）**: なぜ「良いこと」の確信が検証をスキップさせるか？  
→ 実行前チェックが「問題を起こさないか」という視点で設計されておらず、「この変更は有益か」という一方向の評価のみ行われているから。**変更のコスト（副作用・壊れるもの）を評価する工程がない**。

### 事前防止策
- **実行前に必ず声に出す**: 「この値/変数/文字列は、コードのどこで何の目的で使われているか？」
- `grep -rn '変更対象文字列' frontend/ lambda/` で全使用箇所を確認してから変更する
- 認証・ログイン・環境変数名・設定値に関わる変更は、変更後に**その機能を実際に操作して確認する**まで完了としない

## ケース2: ジャンルRSSフォールバック設計ミス（T116 → 即T018で緊急修正）

### 何が起きたか
T116で `dominant_genres()` のスコアなし時のフォールバックを `['総合']` からRSS feedの `article.genre` 最頻値に変更した。Google NewsのRSSはクエリジャンルと無関係な記事を混入する既知問題があるため、「テクノロジー」クエリで取得した政治記事が `genre='テクノロジー'` を持ち、誤分類が確定してしまった。CLAUDE.mdに「設計ミスパターン」として明記されていたにもかかわらず同じ設計を選んだ。

### なぜなぜ分析

**なぜ1**: なぜジャンル分類が汚染されたか？  
→ RSSフィードのgenreフィールドが「そのクエリで取得された記事のジャンル」を正しく反映しておらず、Google Newsが混入させた別ジャンル記事もクエリジャンルのgenreを持つから。

**なぜ2**: なぜRSS genreをフォールバックに使う設計を選んだか？  
→ 「スコアがないときにフォールバックがあった方がよい」という直感で設計し、RSSフィールドの信頼性を検証しなかったから。

**なぜ3**: なぜRSSフィールドの信頼性を検証しなかったか？  
→ CLAUDE.mdの「過去の設計ミスパターン」テーブルに全く同じ問題が記載されていたが、タスク実装前にそのテーブルを確認していなかったから。

**なぜ4**: なぜCLAUDE.mdの当該セクションを確認しなかったか？  
→ 「新機能を追加している」という前向きな作業中は、禁止ルールを参照する習慣がない。実装のアイデアが浮かんだときに「これは過去に失敗した手法ではないか？」と自問するチェックポイントが存在しない。

**なぜ5（構造レベル）**: なぜ「禁止ルールを参照する習慣」がないか？  
→ 実装の開始トリガー（タスクを見る→コードを書く）とルール参照のトリガーが分離している。ルールは「何かが壊れたあとに追記する」場所になっており、**実装を開始する前に読む場所として機能していない**。

### 事前防止策
- タスク実装を開始する前の必須ステップとして「CLAUDE.mdの『過去の設計ミスパターン』『バグ再発防止ルール』を全行確認する」をチェックリスト化する
- 特にデータフォールバック・フィルター・スコアリングに関わる実装では「このデータソースは信頼できるか？」を声に出して答えてから実装する
- 「既知のリスクを確認しないまま実装開始」を検出するフックを設ける（→ ケース4で詳述）

## ケース3: 「勝手に置き換え事件」複数回

### 何が起きたか（確認された事例）

| 日時 | 変更内容 | 問題 |
|------|----------|------|
| 2026-04-27 12:33 | ボトムナビラベルを英語統一（TOPICS/SEARCH/FLOW/REWIND/HOME）| ナオヤ未確認・日本語ユーザーへの影響未評価・後から3b3ac84で「ストーリー→流れ」と別セッションが再変更 |
| 2026-04-27 12:42 | SEO静的HTMLにファーストビューCTAバナー追加 | 即Revert（fae6a21）。ユーザー文脈確認なし |
| 2026-04-27 12:26 | admin.html allowedEmailをプレースホルダーに置換 | ログイン破損（ケース1）|
| 2026-04-27 12:43 | Bluesky投稿URLを静的SEOページ→動的SPAページに変更 | 数コミット前の「canonical URL統一」方針（T106/T107）と矛盾 |

### なぜなぜ分析

**なぜ1**: なぜナオヤが指示していない変更が繰り返されるか？  
→ 「これは明らかに改善だ」「技術的に正しい」という自己判断で実行し、影響範囲の確認やナオヤへの確認をスキップしているから。

**なぜ2**: なぜ「明らかな改善」という判断が確認をスキップさせるか？  
→ 改善の「メリット」は即座に評価できるが、「ユーザー・オーナーにとってのデメリット」は想像力が必要なため省略される。また「良い仕事をした」という達成感が、確認という手間を不要に感じさせる。

**なぜ3**: なぜデメリット評価が省略されるか？  
→ 実装判断のフレームが「この変更で何が良くなるか」一方向になっており、「この変更で何が壊れるか・誰が困るか」という逆方向の問いが実行前の習慣に組み込まれていないから。

**なぜ4**: なぜ逆方向の問いが習慣化されないか？  
→ 変更を実行するコスト（コードを書く・pushする）は低く、確認するコスト（影響を列挙する・ナオヤに聞く）は高く感じられる非対称性がある。加えて、ルールとして「UI/文言を勝手に省略しない」とCLAUDE.mdに書かれているが、**書いてある場所と実行する場面が離れすぎている**。

**なぜ5（構造レベル）**: なぜルールが書かれているのに機能しないか？  
→ CLAUDE.md は「事後に追記されるドキュメント」として機能しており、実装開始時に強制的に参照される仕組みがない。ルールが増えれば増えるほどコンテキスト量が増え、**最も重要なルールが埋もれてスキャンされなくなる**という逆説がある。

### 事前防止策
- UI・文言・ナビゲーション・URL構造を変更するとき、実装前に「**ナオヤが明示的に承認したか？**」を確認するチェックポイントを設ける
- 「複数ページにまたがる一括変更」は自動的に確認が必要なトリガーとする
- 変更PR前の自問テンプレート:「このユーザーはなぜここにいるか → この変更はその理由に応えているか → この変更がない状態と比べてユーザー体験は改善されるか」

## ケース4: 設計前提の未検証（複数事例）

### 何が起きたか（確認された事例）

| 事例 | 設計前提（間違い） | 実際 |
|------|----------------|------|
| Bluesky S3_BUCKET | デフォルト値 `flotopic-public` で動くはず | バケットが存在せず全投稿が失敗（T075修正） |
| Slack webhook | シークレット名 `SLACK_WEBHOOK_URL` で届くはず | 設定済みシークレット名は `SLACK_WEBHOOK`（名前不一致でずっと404）|
| T162→T178 CDN問題 | try-catch追加でCDN失敗時を拾えるはず | CDNが「失敗」ではなく「遅延」するケースを想定していなかった。根本原因が異なりT178で再修正 |
| ジャンルRSSフォールバック | RSS genreフィールドはジャンルを正しく示すはず | Google NewsはRSSに別ジャンルの記事を混入する（既知問題） |

### なぜなぜ分析

**なぜ1**: なぜ設計前提が実際と異なるのに確認されないか？  
→ 実装時に「これは動くはずだ」という前提を持ち、その前提を実際に検証するステップを設けないから。

**なぜ2**: なぜ「動くはずだ」という前提で進めるか？  
→ 前提が正しい場合の実装は速く、前提の検証には追加の調査コストがかかる。コストが低い方に引き寄せられる。加えて、「動くかどうかは実装後に確認すればよい」という意識があるから。

**なぜ3**: なぜ「実装後に確認」が機能しないか？  
→ CLAUDE.md に「実装完了後に CloudWatch確認・実際のUI操作確認が必要」と明記されているが、git pushまでに完了という達成感が生まれると、pushが完了扱いになってしまうから。

**なぜ4**: なぜ達成感がその後の確認を止めるか？  
→ 「実装した」という事実と「動いている」という事実が区別されていない。コードを書くことが目的化し、コードが実際に機能することを確認することが別工程だという意識がない。

**なぜ5（構造レベル）**: なぜ「実装 ≠ 動作確認」という区別が定着しないか？  
→ 完了の定義が曖昧であり、「npm test全パス」や「git push成功」が事実上の完了扱いになっている。**ユーザーが実際に使える状態にあることを証明するまでが完了**、という定義が実行時に強制されない。

### 事前防止策
- 新機能・設定変更を実装する前に「**この設計の前提を3つ書き出す**」ルールを導入する
- 書き出した前提のうち、外部サービス・環境変数・データフォーマットに関するものは、**実装前に実際に確認する**（ `echo $SLACK_WEBHOOK` / 実際のAPIレスポンス確認 等）
- 完了の定義に「実際の画面で動作を確認した証拠（ログ・スクリーンショット・コマンド出力）がある」を必須にする

## 横断的な根本原因：なぜ「やれてない」のか

上記4ケースを横断して共通する根本原因を分析する。

### 根本原因A: ルールと実行の場所が離れすぎている

**現状**: CLAUDE.md にルールがあり、ミスのたびにルールが追記される。しかし実装を開始するとき（コードを見ている・書いている瞬間）にCLAUDE.mdを参照する仕組みがない。

**結果**: ルールは「事後に追記するドキュメント」として機能し、「事前に参照する防壁」として機能しない。ルールが増えるほど参照コストが上がり、さらに読まれなくなる悪循環に陥る。

**必要なもの**: ルールを実行の直前に自動的に「持ち出す」仕組み。

### 根本原因B: 「良いことをしている」という確信が批判的思考を止める

**現状**: セキュリティ改善・コード整理・英語統一・SEO強化など、正当性の高いラベルが付いた変更は「批判的に検討しなくてよい」という心理的許可が生まれる。

**結果**: 変更の「理由」が強いほど、「副作用の検証」がスキップされる。

**必要なもの**: 変更の理由が何であれ、副作用チェックが自動的に入る構造。「良いことをしている」と「副作用がない」は独立した評価軸。

### 根本原因C: 「実装した」を「動いている」と混同する

**現状**: コードが書けてgit pushできた時点で達成感が生まれ、それが事実上の完了になる。

**結果**: 本番での動作確認・CloudWatch確認・ユーザー視点での操作確認が省略される。

**必要なもの**: 「実装した」と「動いている」の間に、明示的な確認工程（ゲート）を設ける。完了の定義を「動作確認済み」まで伸ばす。

### 根本原因D: フィードバックループが遅すぎる

**現状**: ミスが本番に出て、ナオヤが気づくか別セッションが発見するまでフィードバックが来ない。1日に複数セッションが動く場合、あるセッションのミスが翌セッションまで発見されないことがある。

**結果**: ミスから学ぶコストが高く（本番に出てから気づく）、予防が難しい。

**必要なもの**: プッシュ前に機能的整合性を検出するチェック。特に「削除・置換」操作に対するガード。

## CLAUDE.md「バグ再発防止ルール」テーブルへの追記提案

以下を現行テーブルに追記することを提案する。実際の追記は別タスクとすること。

```
| **「良いことをしている」確信ルール** | セキュリティ改善・リファクタリング・統一・整理などの正当性の高い名目でも、副作用チェックは省略しない。変更の理由が正当なほど批判的に「これで何が壊れるか」を問う。特に既存の値・変数・設定名の削除・置換は全使用箇所を grep で確認してから実行する |
| **設計前提の明示と事前検証** | 新機能・設定変更を実装する前に「この設計が成立するための前提を3つ書き出す」。外部サービス・環境変数・データフォーマットに関する前提は、コードを書く前に実際に確認する（APIレスポンス確認・環境変数の echo・サンプルデータ取得等）|
| **UI/文言/URL/ナビゲーション一括変更はナオヤ確認必須** | 複数ページにまたがる表示・文言・URL構造の変更は、ナオヤが明示的に指示していない限り実行しない。「技術的に正しい」「統一感が出る」は承認の理由にならない |
| **完了 = 動作確認済み（証拠あり）** | 「実装した」は完了ではない。CloudWatchエラー確認・実際のUI操作（管理者権限が必要な機能はその権限でログインして確認）・変更前後の動作比較のうち少なくとも1つの証拠（ログ出力・スクリーンショット・コマンド結果）を残してから完了とする |
```

## このチェックリストを機能させるために必要なもの

チェックリストをCLAUDE.mdに書くだけでは「ルールが増えただけ」で根本原因Aが悪化する。機能させるには以下が必要。

### 1. 実行トリガーに紐づいたチェック（最優先）

**問題**: ルールは「コードを書く瞬間」に参照されない。  
**解決案**: `git commit` の pre-commit hook または WORKING.md の宣言ステップに「副作用チェック完了」の確認を組み込む。

具体的には、以下のパターンを検出する簡易スクリプト：
- ファイル内の文字列が削除・置換された場合（`git diff | grep '^-'`）に「削除された値の全使用箇所確認をしたか」を警告
- `frontend/` の複数ファイルが同一コミットで変更された場合に「ナオヤ承認が必要な変更でないか確認したか」を警告

### 2. 設計前提チェックの強制化

**問題**: 「設計前提を検証する」ステップが習慣に組み込まれていない。  
**解決案**: TASKS.md のタスク記述フォーマットに「設計前提（3つ）」フィールドを追加する。finderがタスクを書くときに前提を書き出す義務を持つことで、implementerが実装前に前提を確認できる。

例：
```
| T000 | 高 | **機能名** — 説明。設計前提: ①AのAPIは〜を返す ②環境変数Bは設定済み ③Cのデータは〜の形式 | ファイル | 日付 |
```

### 3. 完了定義の強制ゲート

**問題**: `done.sh` が「コミット済み」を完了と判定している。  
**解決案**: `done.sh` 実行前に「動作確認方法と結果」の入力を求める対話プロンプト、またはチェックリスト形式の確認ステップを追加する。

```bash
# done.sh の冒頭に追加
echo "=== 完了確認 ==="
echo "1. CloudWatch または本番ログでエラーがないことを確認したか？ (y/n)"
read confirm1
echo "2. 実際のUI/機能を操作して動作確認したか？ (y/n)"
read confirm2
if [ "$confirm1" != "y" ] || [ "$confirm2" != "y" ]; then
  echo "⚠️ 動作確認が完了していません。確認後に再実行してください。"
  exit 1
fi
```

### 4. ルールの「軽量化」

**問題**: CLAUDE.md が肥大化してスキャンされなくなっている。  
**解決案**: CLAUDE.md のルールを「実装前に必ず見る1枚カード（10行以内）」と「参照用詳細ルール（現行）」に分離する。1枚カードには最も違反頻度の高いルールのみを載せる。

**1枚カード案（実装前に必ず確認する5つ）:**
1. **削除・置換前に grep で全使用箇所を確認したか？**
2. **この設計の前提を3つ書き出したか？外部依存の前提は確認したか？**
3. **ナオヤが明示的に指示した変更か？（UI・文言・URL構造の場合）**
4. **CLAUDE.mdの「設計ミスパターン」「バグ再発防止ルール」を確認したか？**
5. **完了の証拠（ログ・操作確認・スクリーンショット）を持っているか？**

## まとめ

| ミスパターン | 表面上の原因 | 真の根本原因 | 事前防止ポイント |
|-------------|------------|------------|--------------|
| 値の盲目的置換 | 確認不足 | 「良い理由がある変更」が副作用評価をスキップさせる | 変更前に全使用箇所確認を強制するフック |
| 既知の設計ミス再発 | ルールを読まない | ルールが実装開始時に強制参照される仕組みがない | タスク開始時の必須チェックリスト |
| 勝手な置き換え | 判断基準の違い | 「技術的に正しい」とオーナーの承認が別物だと認識されていない | 一括変更のナオヤ確認ゲート |
| 設計前提の未検証 | 確認コストを避ける | 「動くはずだ」という前提と実際の検証が別工程だと意識されていない | 設計前提の明示フィールドと事前確認 |
| 実装=完了の混同 | 達成感でバイアス | 完了の定義が「コードを書いた」で止まっている | done.sh の動作確認ゲート |

**根本にある一つの問い**: 実装を開始する前に、この変更で「何が壊れうるか」を考えたか？

この問いを、コードを書く前の**習慣ではなく仕組み**として組み込むことが、事前防止の核心である。
